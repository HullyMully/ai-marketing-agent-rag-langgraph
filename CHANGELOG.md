# Changelog

All notable changes to this project are documented here. Format loosely follows
[Keep a Changelog](https://keepachangelog.com/) and Semantic Versioning.

## [Unreleased]
### Docs & tests
- **Repositioned as a "Configurable AI Customer Assistant Platform"** across
  `pyproject.toml`, the README (EN/RU) and all docs. Rewrote `docs/langgraph-flow`
  and `docs/api` (EN/RU) to the `plan → act` planner + backend-validation flow,
  documented the planner input/output JSON, lead/ticket rules, RAG usage, memory,
  DeepSeek/OpenAI-compatible setup and `.env`/company-config safety, and refreshed
  the demo walkthrough, limitations, portfolio case study and the flow diagram.
- **Tests:** added a mocked planner-reply passthrough test and scenario coverage
  for casual/abusive input, the confused-user memory flow, single-lead creation,
  escalation and RAG sources — all with a mocked planner/LLM. Added an optional
  real-LLM integration test (`integration` marker, skipped unless
  `RUN_LLM_INTEGRATION=1`). Fixed the demo-asset manifest to match the real PNG
  screenshots so the asset check passes.

### Removed / Cleaned up
- **Dropped the scripted "repeated confusion → ticket" escalation.** Ordinary
  confusion, jokes or venting (e.g. "what do you mean?", "I don't remember",
  "another one", swearing) now switch to gentle exploration instead of looping a
  menu or opening a fake ticket. Removed `unclear_request` from the ticket
  justification rules entirely.
- **Hardened ticket validation against trivial escalation:** a bare greeting,
  "I am a new customer" or small talk can no longer open a ticket even if the
  planner asks for one with high confidence — only an explicit human request,
  genuine frustration, or a custom/enterprise need (or a high-confidence
  *complaint/enterprise* reason) qualifies.
- **Lead/ticket confirmations are now LLM-generated in real mode** and only
  produced *after* the backend has actually created the record, so the assistant
  never claims success before it happens. Templated confirmation lines remain
  only as offline (`MOCK_LLM`) fallbacks.

### Changed (web demo UI)
- The right-hand panel now surfaces the **planner's recommended action**, the
  **backend validation result** (approved / blocked + reason) and whether the
  **action was executed**, next to the existing real lead-draft, created-lead,
  created-ticket and knowledge-source cards. No fake CRM/ticket cards: empty
  states are honest, and real records are shown only once the backend creates
  them. Quick-action buttons still only send a normal message to `/chat`.

### Added
- **Pydantic planner schema** (`PlannerOutput` with enums for `user_intent`,
  `assistant_mode` and `recommended_action`, plus `budget_unknown` and
  `user_agrees_to_proceed` in `extracted_fields`). The LLM is forced to emit
  JSON only; output is parsed safely, validated with Pydantic, and **repaired
  with one retry** if invalid. If it still fails, the planner returns a
  controlled internal error (no crash, no canned phrase) and the backend writes
  the reply.
- **Backend action-validation layer** (`app/agent/validation.py`):
  `validate_lead_creation` and `validate_ticket_creation` are the single source
  of truth for whether a recommended action runs. Leads require name, company, a
  valid email, a service interest and a budget (or an explicit
  budget-unknown + user agreement), block duplicates, and only run when the
  planner recommended `create_lead`. Tickets only open for genuine human
  requests / complaints / enterprise review, never for greetings, jokes,
  ordinary confusion ("what do you mean?", "I don't remember", "I told you") or
  general questions.
- When validation rejects an action (or the planner errors), the **final reply
  is generated by the LLM** from the validation reason and missing fields — no
  hardcoded fallback text in normal chat (the deterministic phrasing is used
  only in offline `MOCK_LLM` mode).
- `/chat` metadata now exposes `planner_decision`, `validation`
  (allowed / reason / missing fields) and `action_executed`.
- Mocked tests for planner JSON parsing/validation, JSON repair, the controlled
  planner error, lead/ticket validation success+failure cases, and `/chat`
  lead/ticket guards (`tests/test_validation.py`).

### Changed
- **Refactored the assistant around an LLM dialogue planner** (`app/agent/planner.py`).
  Each turn the planner receives the company profile, relevant RAG knowledge,
  recent history, session memory, the lead draft, ticket state and available
  actions, and returns one structured JSON decision (intent, mode, extracted
  fields, recommended action, reply, sources, confidence). The graph is now a
  thin `plan → act` orchestrator and the **backend validates and executes**
  actions — the LLM never creates a lead or ticket on its own. The old
  keyword-routing graph is gone; its deterministic behaviour now lives in the
  offline mock planner used for `MOCK_LLM` mode and as the safe fallback.
- `/chat` now also returns `assistant_reply`, `user_intent`, `recommended_action`
  and `knowledge_used`; the web panel shows mode, intent, known facts, lead draft,
  missing fields, last action and sources.
- LLM client hardened for OpenAI-compatible / DeepSeek endpoints: configurable
  timeout/temperature, single retry, safe error handling and logging without
  secrets.

- Repositioned the project as a **configurable AI customer assistant platform**.
  The company identity (name, domain, description, contact, assistant name,
  escalation target, industry) is now loaded from `config/company.example.json`,
  an optional git-ignored `config/company.local.json`, and environment variables.
- Removed the hardcoded demo company from the product UI, agent and core code.
  The web UI shows the configured brand, generic suggested prompts and a real
  "Conversation state" panel driven by backend metadata.
- Knowledge base and sample data ship as a clearly-labelled fictional sample
  using `.example` domains; documents are meant to be replaced per deployment.

### Added
- `docs/company-configuration.md` (+ RU) describing how to configure a real
  company, replace the knowledge base and run ingestion.
- `scripts/test_production_flow.py` manual end-to-end check against a local server.
- Natural-dialogue policy with per-session dialogue state (greeting / refusal /
  confusion / frustration counters, `exploration_mode`, `qualification_paused`).
  The assistant now answers social greetings, jokes, confusion and frustration,
  never repeats the same qualification question more than twice, and switches to
  an exploration mode (general guidance, no company/budget/email) when the user
  can't or won't share details — pausing qualification after repeated refusal.
- Conversation **mode** (answering / exploring / qualifying / paused), known
  interests and a suggested next step are returned by `/chat` and shown in the
  web "Conversation state" panel; a lead draft is only shown while qualifying.
- Service interest now combines additively ("and SEO" → "Paid ads + SEO"), and a
  lead is created only when the user clearly wants to proceed and all fields are
  known.

## [0.1.0] – 2026-06-22
### Added
- FastAPI backend: `/health`, `/chat`, `/crm/leads`, `/tickets`, `/knowledge/ingest`, `/metrics/demo`.
- LangGraph stateful agent with intent classification, RAG retrieval, action routing,
  lead collection, tool calls and human escalation.
- LangChain prompt templates and an LLM abstraction with a `MOCK_LLM` offline mode.
- RAG pipeline: markdown loader, recursive chunking, embeddings (OpenAI-compatible
  or deterministic mock) and Qdrant vector store with an in-memory fallback.
- Synthetic sample knowledge base for a fictional digital marketing studio.
- Mock CRM, support tickets and session memory backed by SQLite + a repository layer.
- Telegram bot (aiogram) talking to the API.
- Demo metrics endpoint.
- Docker Compose (app + Qdrant, optional bot), Dockerfile.
- pytest suite and full documentation set.
